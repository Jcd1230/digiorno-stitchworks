# Designer 1 Format Notes

These notes document the file formats currently generated by `designer1-rs` and the byte layouts that were verified against a known-good Husqvarna/Viking Designer 1 disk image.

The `SHV` format remains partly empirical. The `PHV` and `MHV` structures below are based on a combination of:

- direct byte comparison against a working exported disk,
- successful on-machine testing,
- and the current implementation in [`src/disk.rs`](src/disk.rs).

## Disk Layout

A working single-menu disk currently contains:

```text
MENU_SEL.PHV
MENU_01/
  DES01_01.SHV
  DES01_02.SHV
  ...
  MENU_01.MHV
MENU_02/
  MENU_02.MHV
MENU_03/
  MENU_03.MHV
MENU_04/
  MENU_04.MHV
```

Notes:

- The machine expects all four `MENU_0X` directories to exist.
- `MENU_01` contains up to 6 design files in current tooling.
- `MENU_02` through `MENU_04` may be placeholder menus, but the directories and `.MHV` files still need to exist.

## PHV: `MENU_SEL.PHV`

`MENU_SEL.PHV` is the root menu file. It defines:

- the disk title,
- labels for `MENU_01` through `MENU_04`,
- a root-menu touch/hitbox metadata block,
- and a 4bpp preview bitmap.

Verified layout:

```text
0x0000..0x0055  86-byte signature/notice region
0x0056..0x0059  4-byte root config/flags
0x005A..0x01D9  16 x 24-byte disk title slots
0x01DA..0x01DB  2-byte constant: 5A A0
0x01DC..0x029D  Menu 1 block: 2-byte menu metadata + 16 x 12-byte labels
0x029E..0x035F  Menu 2 block: 2-byte menu metadata + 16 x 12-byte labels
0x0360..0x0421  Menu 3 block: 2-byte menu metadata + 16 x 12-byte labels
0x0422..0x04E3  Menu 4 block: 2-byte menu metadata + 16 x 12-byte labels
0x04E4..0x04F8  21-byte root-menu hitbox block
0x04F9..0x04FA  Bitmap header: [height] [width]
0x04FB..EOF     Nibble-packed 4bpp bitmap
```

Current verified constants:

```text
root config:           00 01 00 00
menu constants:        5A A0
menu 1 metadata:       39 82
menu 2 metadata:       39 5C
menu 3 metadata:       39 36
menu 4 metadata:       39 10
root bitmap size:      height=0xB5 (181), width=0xAD (173)
root bitmap offset:    0x04FB
```

Verified root-menu hitbox block:

```text
01 1A 78 1D 17
01 1A 52 1D 17
00 1A 2C 1D 17
00 1A 06 1D 17
00
```

This is currently treated as a fixed 21-byte structure. Matching the known-good bytes here was required for the generated disk to work correctly on the machine.

### PHV label encoding

- The disk title section stores 16 copies of the same title, each padded/truncated to 24 bytes.
- Each menu label section stores 16 copies of the same menu label, each padded/truncated to 12 bytes.
- Labels are ASCII-ish:
  - printable ASCII and spaces are preserved,
  - unsupported bytes are replaced with `_`,
  - unused bytes are `0x00`.

## MHV: `MENU_0X/MENU_0X.MHV`

Each `MHV` file defines a submenu screen. For `MENU_01`, this is the design grid. For `MENU_02` through `MENU_04`, the current exporter writes placeholder screens.

Verified layout:

```text
0x0000..0x0055  86-byte signature/notice region
0x0056..0x0115  16 x 12-byte menu label slots
0x0116..0x0117  2-byte constant: 77 FB
0x0118          1-byte grid-layout flag
0x0119..0x013C  36-byte file allocation / touch-zone list
0x013D..0x013E  Bitmap header: [height] [width]
0x013F..EOF     Nibble-packed 4bpp bitmap
```

Current verified constants:

```text
submenu constant:      77 FB
grid layout flag:      06
submenu bitmap size:   height=0xEE (238), width=0xF4 (244)
submenu bitmap offset: 0x013F
```

### MHV file allocation list

The 36-byte block at `0x0119..0x013C` maps touch zones to design files.

Current behavior:

- `MENU_01` writes `01, 02, ...` for each populated design slot, then zero-fills the rest.
- `MENU_02` through `MENU_04` currently write all zeros.

Example for a menu with two designs:

```text
01 02 00 00 00 00 ...
```

That means the first two active grid zones refer to `DES01_01.SHV` and `DES01_02.SHV`.

## Shared Bitmap Encoding

Both `PHV` and `MHV` end with a nibble-packed 4bpp image:

- high nibble first,
- then low nibble,
- row-major order,
- palette values are implementation-defined and mostly used for the machine UI preview.

Important:

- Remaining differences versus the known-good `PHV`/`MHV` files are in bitmap content, not in the validated control/header regions.
- The machine accepted the generated files once the structural bytes matched, so these bitmap differences appear cosmetic or at least non-fatal.

## Signature Region

The first 86 bytes of `SHV`, `PHV`, and `MHV` are a signature/notice area.

Known variants:

- `Official`: ASCII notice
  `Embroidery disk created using software licensed from Viking Sewing Machines AB, Sweden`
- `Zero`: 86 zero bytes

For the machine tested during this project:

- `Zero` could trigger a warning that the disk was not made by official software.
- `Official` avoided that warning.

The current default in the app and CLI is `Official`.

## SHV Output Layout

`SHV` remains more empirical than `PHV`/`MHV`, but the current writer uses this structure:

```text
0x0000..0x0055  86-byte signature/notice region
0x0056          one-byte design name length
0x0057..var     ASCII design name
var             6-byte preview header: height, width, h/2, w/2, h/2, w/2
var             4bpp preview bitmap
var             summary block: color count, constants, bbox, total record count
var             14-byte color rows
var             2-byte-record stitch stream
```

Current validation checks:

- summary total records equals stitch stream bytes / 2,
- summary extents equal extents recomputed from the decoded generated stream,
- final decoded position returns to origin.

## Confidence / Open Questions

High confidence:

- `PHV` title/menu label regions
- `PHV` root hitbox block
- `MHV` header and file-allocation list region
- bitmap header offsets and dimensions

Still empirical:

- exact meaning of the `PHV` menu metadata pairs `39 82`, `39 5C`, `39 36`, `39 10`
- exact semantic meaning of each byte inside the 21-byte `PHV` hitbox block
- full `SHV` compatibility rules versus files produced by commercial tools
- whether some bitmap details affect edge-case machine behavior
